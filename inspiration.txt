6. IC信号衰减曲线的关键发现（2026-04-21 factor_ic_test, 200只股票, 2020-2022）
   - 清晰的"均值回归→动量→反转"模式：
     1天(强负IC -0.05) → 5天(正IC +0.03) → 10天(峰值 +0.05) → 15天(+0.03) → 30天(归零) → 60天(强负 -0.05)
   - 1天负IC = 短期均值回归（今天涨明天大概率回落）
   - 5-15天正IC = 中期动量（强势延续）
   - 10天是动量信号峰值，pred_len=10 可能优于 pred_len=15
   - 60天负IC = 长期反转（涨多了要跌回来），解释了v6 pred_len=60效果差的原因

7. 特征信号强弱排序（IC测试验证）
   - 最强：大单/特大单相关（ratio_8特大单买占比 IC=0.042, ratio_14, ratio_7）
   - 中等：订单活跃度（total_buy_order_id_count, trade_count）
   - 最弱：中单相关（ratio_6中单买占比, ratio_9中单卖占比）
   - 结论：大单=聪明钱假说被数据验证，散户指标预测力差
   - 所有ICIR < 0.5（最高0.34），信号存在但不稳定，牛熊切换时IC会反转

8. IC测试的局限性（解读时注意）
   - 日均值压平了日内tick模式，IC是信号下界，CNN能提取更多
   - 只测线性排序相关，非线性组合（如"ratio_8高且ratio_9低"）捕捉不到
   - 收盘价max ≠ tick级max，模型用日内最高价，波动更大信号更丰富
   - 截面IC受股票数影响，<30只时单日IC噪声大

9. 训练过拟合对策（v7前2个fold观察）
   - Val loss U型曲线：最佳epoch在5-8，patience=10太大
   - RankIC前几epoch ~0.05（有效），后半段衰减到0甚至负
   - 已改：patience 10→5，把模型卡在"刚学到信号、还没过拟合"的区间

10. day头从精确天数改为粗分段（降低学习难度）
    - 问题：max_day预测精确哪一天（15分类）太难，val_loss_max_day爆炸性上升

11. day头单独加强正则化（抑制过拟合）[v8 已实施]
    - 问题：max_day在训练集上能"背答案"但不泛化，log_var下降后加权反而拖累总loss
    - 方案：在ViT的day预测头前面单独加Dropout(0.5)，比value头更重的dropout
    - 可配合冻结day头log_var在固定低权重（如1.0），防止"自信地犯错"
    - 好处：不改架构不改loss，只调正则强度
    - 配套：--day_sigma 1.0 → 2.0（PeakDayLoss 目标分布放宽一半）

15. Warm-start 跨 fold 的累积污染假说（v8 验证中）
    - 观察：v7 训练 11 个 fold 后，后期 val loss + RankIC 全面退化
    - 假说：Warm-start 每 fold 都 load 上一 fold 的完整权重（CNN + ViT + heads）
      11 个 fold 叠加后，权重里堆积了各时代的 artifact（节假日/结算日/季节节奏），跨期失效
    - Day 头尤其受害：日期信号弱，模型只能抓时代性特征，不可复用
    - 验证路径：如果 #11（day 头 Dropout）生效、v8 后期 fold 不再发散，
      说明污染可通过 day 头正则化压住，warm-start 本身无需改
    - 若 v8 仍后期退化，考虑 #16

16. Warm-start 只加载 CNN，ViT + heads 每 fold 重新初始化（备选方案）
    - 动机：CNN 学"tick → 日级表征"是通用技能，跨期复用合理
      ViT + heads 学"表征 → 未来标签"的映射，时代特化严重，不该累积
    - 改动：train_rolling.py 加载 checkpoint 时只 load feature_extractor，
      跳过 vit_model.load_state_dict
    - 代价：每 fold ViT 从随机初始化开始，收敛更慢，总训练时间 +20~30%
    - 好处：消除 ViT 层的时代偏置累积
    - 前提：#11 在 v8 被验证仍不够时再做

18. 双卡 V100 利用方式 — 并行跑不同实验 vs 单任务加速
    背景：服务器有 2 张 V100 32GB，想让两张卡都跑起来

    3 种方案对比：

    A. DataParallel（nn.DataParallel）— 不推荐
       - 改动：1 行代码 model = nn.DataParallel(model)
       - 加速：仅 1.3-1.5x（master GPU 过载，GIL 瓶颈）
       - PyTorch 已废弃，浪费硬件

    B. DistributedDataParallel（DDP）— 真正单任务加速
       - 单次训练时间 100h → ~55h，加速 1.7-1.9x
       - 每卡独立处理 batch，all-reduce 同步梯度
       - 改动量大：~100-150 行代码（init_process_group + DistributedSampler +
         每 fold 里处理 rank / barrier）+ 1-2 天调试
       - PyTorch 1.10.1 的 DDP 可用但 API 不如 2.0 优雅
       - 优先级：低。除非单次训练真的成为瓶颈

    C. 并行跑不同实验 — ⭐ 强烈推荐的默认做法
       - GPU 0: v_X（主假说），GPU 1: v_Y（候选假说）
       - 改动：0 行代码，打开两个终端各跑一个
       - 研究效率 2x（同样时间验证两套假说）
       - 不破坏 warm-start 链，不增加复杂度
       - 适用于：模型结构/超参数探索期，需要多方向验证时

    选择 C 的理由：
    - 现在项目阶段缺的是"验证哪个方向对"，不缺速度
    - 如果 100h 后主方案失败，再串行跑下一个又得 100h
    - 并行跑 2 个实验，100h 出 2 组结果

    C 的具体执行（Windows）：
       终端 1：set CUDA_VISIBLE_DEVICES=0 && python -m src.train_rolling --output_dir runs_X ...
       终端 2：set CUDA_VISIBLE_DEVICES=1 && python -m src.train_rolling --output_dir runs_Y ...

    C 的内存预算：
    - 每个进程独立缓存 GlobalDataCache
    - 500 stocks × 1272 days × 100KB × 2 process = ~130 GB
    - 256 GB 服务器够用；紧张时其中一个加 --preload off 回退

    C 的推荐实验搭配（2026-04-24）：
    - v8（day_sigma=2.0 + Dropout(0.5)）vs v9a（pred_len=10，IC 峰值对应）
      两条完全不同的路径：v8 走"强化正则"，v9a 走"换预测目标"
      结果可对比诊断：
        v8 好 → 过拟合问题
        v9a 好 → 预测窗口选错了
        都好 → 两个问题都存在
        都不好 → 信号本身不存在，需要重新审视特征
    - 其他备选对比组：
      v8 vs v9b（sigma=3.0 更宽）：看 sigma 值的敏感性
      v8 vs v9c（weight_decay=1e-2 无 Dropout）：验证是 Dropout 有效还是"任意强正则"有效（v9c 需改代码）
      v8 vs v_ws_cnn（只 warm-start CNN，inspiration #16）：验证是 day 头污染还是累积漂移

    何时升级到 B（DDP）：
    - 单次训练超过 200 小时且关键路径
    - 基线已稳，要做大规模最终训练（而非探索）
    - 迁移到 PyTorch 2.0+ 后顺便做（DDP API 更友好）

19. CSV → Parquet 一次性转换，永久加速数据加载
    - 动机：pyarrow 引擎让 CSV 读取快 2-3x，但 pyarrow 在服务器 Python 3.8 + Win
      装不上（DLL 错误）。换用 Parquet 格式比 CSV 快 5-10x，比 pyarrow 还快，
      不依赖 pyarrow 库
    - 原理：Parquet 是列式二进制格式，类型已保存（不用 to_numeric 推断），
      文件只有 CSV 的 30-50% 大小
    - 实施：
      * 新建 src/convert_csv_to_parquet.py，ProcessPoolExecutor 并行转换
      * 转换时保留 CSV（方案 A），并列存在。Parquet 当性能缓存，CSV 当原始备份
      * dataset.py 里 _read_csvs_to_per_stock_arrays：自动检测是否存在 Parquet，
        有则用 pd.read_parquet，没有则 fallback 到 pd.read_csv
      * 无需额外参数，无需清理 pyc，改完 dataset.py 即可
    - 代价：
      * 一次性转换 ~10-15 分钟（1272 文件 × 8 并行）
      * 磁盘多 ~40%（Parquet ~30-50% CSV 大小，但多了一份）
    - 收益：
      * GlobalDataCache 启动时间从 ~5-10 分钟降到 ~1-2 分钟
      * 每次训练启动都立刻受益（累计节省几小时）
      * 所有读数据的脚本（回测、IC 测试等）一并受益
    - 优先级：锦上添花，不紧急。当前瓶颈是训练算法而非数据 I/O
    - 触发时机：
      * v8 跑完后做（下次 v9 启动就能受益）
      * 并行跑 v8 + v9 之前做（两个进程都享受启动加速）
      * 或者任何时候觉得"每次等启动那几分钟有点烦"
    - 可选进阶：转换后可在新服务器（PyTorch 2.0+）用 pyarrow 引擎的 parquet 读取，
      对大数据集还能再快 2-3x

17. 断点续训（fold 级自动跳过）
    - 动机：v8 规模 ~19 小时（77 fold × ~15min），长训中途崩溃损失大
      当前优雅停止 + fold 间 checkpoint 已覆盖 80% 场景，但 fold 内崩溃无恢复
    - 最简方案（约 15 行代码）：重跑同一条命令时，自动识别已有 model_final.pth
      的 fold 并跳过，从下一个未完成 fold 开始。不需要新参数。
    - 实现要点：
      * 循环里 if os.path.exists(fold_dir/model_final.pth): skip + load as previous_model_path
      * 从 checkpoint 里读 best_val_loss / best_epoch 恢复 summary 表
      * Config 一致性检查：checkpoint 里的 args 与当前 args 不一致时报警
      * torch.load try/except：文件损坏时重训该 fold
    - 覆盖场景：人为 Ctrl+C 继续、机器重启、fold 内 OOM/崩溃、跨设备迁移
    - 不覆盖（复杂版才需要，暂不做）：fold 内 epoch 级续训、TB 日志合并
    - 判断时机：
      * v8 先不加（v7 证明单 fold 稳定，跑一次 19h 能接受）
      * 若 v8 或 v9 真的崩过浪费了几小时 → 立即加最简版
      * 若后续需要持续迭代（每天都跑）→ 提前加

20. Warm-start 矛盾分析：为什么 LLM 能保留权重，我们的 ViT 不能？（2026-04-26）
    - 核心矛盾：大模型预训练都保留历史权重，但我们 ViT warm-start 导致 regime shift 累积污染
    - 原因：LLM 预训练的三个前提条件在我们场景全部不满足
      * 分布平稳性：语言几十年不变 vs 市场 regime 每几个月就变（牛熊/政策/板块轮动）
      * 数据/参数比：LLM 300B tokens/175B params vs 我们 12K 样本/50.6M params（严重过参数化）
      * 训练顺序：LLM shuffle 训练 vs 我们必须按时间滚动（连续暴露于同一 regime）
    - 参数分布：CNN 1.76M (3.4%) vs ViT 50.6M (96.6%)
      小数据 + 非平稳分布 + 高容量 = warm-start 必然累积 regime 记忆
    - 解法方向（按优先级）：
      A. 降低 ViT 容量（最推荐，与 #14 呼应）
         * embed_dim 1024→256, depth 4→2，参数从 50M 降到 ~3-5M
         * 模型容量匹配数据量后，没有能力记忆 regime 特异性，warm-start 可能重新 work
         * 这和 LLM 的逻辑一致：不是靠丢弃权重，而是靠让容量匹配数据
      B. 差异化学习率
         * CNN 正常 lr，ViT 用 1/10 lr warm-start（类似 LLM fine-tune 小 lr 策略）
         * 简单易实现，可与方案 A 组合
      C. 全历史预训练 + fold 微调
         * 先用全部数据训一个"基模"ViT，每 fold 只微调（冻结大部分层）
         * 类似 LLM pretrain → finetune 范式
      D. EWC（弹性权重巩固）
         * 惩罚对"旧 fold 重要参数"的修改，保护通用知识，允许适应新 regime
         * 实现复杂度高，效果不确定
      E. Warm-start backbone，仅重置 heads（折中方案）
         * ViT 4层 Transformer 可能学了通用序列关系，heads 过拟合了 label 分布
         * 只重置 heads（33K 参数），保留 backbone（50.5M）
    - 实验路径：
      * v9 (cnn_only) 先验证"ViT 重置能稳定"
      * 若 v9 成功 → 下一步试方案 A（降容量 + warm-start），看能否两全
      * 方案 B（差异化 lr）可同时试，成本极低

21. v10 模型框架设计：降 ViT 容量 + 全市场训练 + 双卡实验（2026-04-27）

    背景：
    - v9 (fold 18+) 验证 cnn_only warm-start 完全解决累积污染
    - 但 RankIC 不稳定（-0.06 ~ +0.14），根因：52.4M 参数 vs 14,500 样本 = 3600:1
    - 需要同时降模型容量 + 增加数据量

    当前参数分布：
    - 1D-CNN: 1.76M (3.4%)  — tick→daily 特征，与股票数量无关
    - StockViT: 50.6M (96.6%) — 序列→预测，过参数化
    - Heads: 33K (~0%) — 预测头

    sample_stride 对样本数的影响（容易遗漏！）：
    - stride=10: 每股每fold 29 样本（v7-v9 用的）
    - stride=5:  每股每fold 58 样本
    - stride=1:  每股每fold 286 样本（10倍，但99.4%输入重叠）

    数据量估算（train_days=480, seq_len=180, pred_len=15, stride=10）：
    | 股票数 | 样本/fold | 全量内存/fold |
    | 500  (v9)  | 14,500   | ~27 GB  |
    | 800  (30/31)| 23,200   | ~39 GB  |
    | 2000 (全A)  | 58,000   | ~98 GB  |

    模型容量推荐（目标参数/样本 = 1:5 ~ 1:20）：
    | 场景       | embed_dim | depth | heads | ViT参数 | 总参数 | 比例(stride=10) |
    | 800只      | 384       | 3     | 4     | 5.3M   | ~7M   | 1:3 → 配stride=5 |
    | 2000只     | 384       | 3     | 4     | 5.3M   | ~7M   | 1:8 ✅           |
    | 2000只+大  | 512       | 4     | 4     | 12.7M  | 14.5M | 1:4 → 需正则化   |
    注：如果 stride=1，样本×10，所有方案比例都很健康

    embed_dim 选择依据：
    - 必须能被 num_heads 整除（256/384/512/768/1024 对 4 heads 都OK）
    - 384 = DeiT-Small 级别，head_dim=96，attention 表达力足够
    - 512 = ViT-Small 级别，head_dim=128，适合大数据量
    - 1024 = 当前，对我们的数据量明显过大

    推荐 v10 框架（2000只全A股）：
    - 1D-CNN: output_dim=512（原1024→512），其余不变，~1.76M
    - StockViT: embed_dim=512, depth=4, num_heads=4, ~12.7M
    - 或保守版: embed_dim=384, depth=3, ~5.3M
    - Heads: 不变（Linear→1 for value, Dropout+Linear→pred_len for day）
    - 总参数: 14.5M（512版）或 7M（384版）

    训练策略调整：
    - num_stocks: 500 → 全部合格股票（~2000）
    - step_days: 10 → 20（减半fold数，相邻fold 98%重叠是浪费）
    - batch_size: 32 → 64（样本多，大batch更稳定）
    - sample_stride: 保持10（或尝试5增加样本）
    - warm_start_mode: cnn_only（v9 已验证有效）
    - preload: off（2000只×480天≈98GB，每fold单独加载）
    - train_days: 480（不够可提高到600）

    双卡 V100 实验设计（方案C：两卡跑不同实验，零代码改动）：
    - 卡0: 全A股 2000只, embed_dim=512, depth=4 (14.5M)
    - 卡1: 30/31前缀 800只, embed_dim=384, depth=3 (7M)
    - RAM: 39+98≈137GB，256GB够用
    - 同时出结果，直接对比
    - 等确定最优配置后再上 DDP 加速

    训练时间估算（单V100, step_days=20）：
    - 2000只: 58K样本, bs=64 → 906 steps/epoch × 8epoch ≈ 7248 steps/fold
      ~0.08s/step → 每fold ~10min, 34 folds → ~5.7h
    - 800只: 23K样本, bs=64 → 363 steps/epoch × 8epoch ≈ 2900 steps/fold
      每fold ~4min, 34 folds → ~2.3h
    - 两张卡同时跑：最多 ~6h 出两组结果


22. 价格通道预测（替代四头方案）
    来源：v9 训练分析，loss_max_day 逐fold爆炸(0.7→1.8)，max_day 头可能是伪任务
    
    核心思路：
    - 用 2 个序列头替代原来的 4 个标量头
    - head_daily_max: Linear(embed_dim, pred_len) → 预测15天逐日最高价收益率
    - head_daily_min: Linear(embed_dim, pred_len) → 预测15天逐日最低价收益率
    - max_value / min_value / max_day / min_day 全部从序列推导，不再需要单独的头
    
    监督信号：4 → 30（+7.5x），每天的daily_max和daily_min都是独立标签
    
    价格通道评估指标：
    - 包络率: 实际价格落在 [pred_min, pred_max] 内的天数 / 15，目标 >90%
    - 通道宽度: mean(pred_max - pred_min)，越窄越有交易价值
    - 突破方向: 价格突破上轨 vs 下轨比例，判断模型偏保守还是激进
    
    数据来源：target_seq[:, 2, :].max(axis=1) 和 .min(axis=1) 已有，不需要新数据
    
    优势：
    1. 监督信号密度提升7.5倍，缓解 "180个token只产生4个信号" 的稀疏问题
    2. 逐日预测天然包含时序结构，模型被迫学习每天的动态变化
    3. 回测可视化直观：预测上下轨包住实际股价线
    4. 消除 max_day 这个难学的分类任务（从序列 argmax 推导即可）
    
    实施计划：v10 先用原4头跑基线，确认参数缩减有效后，v11 切换到价格通道方案
